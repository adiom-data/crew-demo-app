# crew-demo-app ("On-board") ŌĆö agent guide

Partner onboarding & management portal on the `adiom-data/framework` stack: a **Bazel-only**
monorepo with a Go + Connect-RPC API/auth server, a Postgres datastore (goose migrations), and a
Vite/React SPA served in production behind a component **gateway**. It is a fork of
`adiom-data/sample-app` renamed to `crew-demo-app`. See [PRD.md](PRD.md) for the product.

**Naming gotchas (read first):**
- The **proto package is still `sample.v1`** and the Go module is `github.com/adiom-data/crew-demo-app`.
  The partner feature lives under `proto/sample/v1/partner.proto` ŌĆö there is no `partner.v1` package.
  Two services share that package: `PartnerService` (authenticated) and `OnboardingService` (public).
- Kubernetes resources use the **`crew-demo-`** prefix (`crew-demo-api`, `crew-demo-postgres`,
  `crew-demo-auth`); container images use **`crew-demo-app-`** (`ghcr.io/adiom-data/crew-demo-app-api`).
  These two prefixes are intentionally different ŌĆö don't "normalize" them.
- `PRD.md` and `prd.md` are the **same file** on macOS (case-insensitive FS); the canonical name is
  `PRD.md`.

## Build / test / run
Bazel is the only build tool. Quick reference (details in [docs/operations.md](docs/operations.md)
and [docs/testing.md](docs/testing.md)):
- Build everything: `bazel build //...`
- Test everything: `bazel test //...` (Go units + Vitest via `//web:test`)
- Regenerate protos after editing `proto/**`: `buf generate`
- Refresh Go BUILD files: `bazel run //:gazelle`
- Seed a local DB: `PGHOST=ŌĆ” ŌĆ” bazel run //cmd/seed`
- Local end-to-end run (Postgres + API + Vite): see [docs/operations.md](docs/operations.md#local-development).

## Code organization principles
Non-obvious conventions that will bite you otherwise:
- **Protos are generated by buf, not Bazel.** `buf generate` writes `gen/go/**` and `web/src/gen/**`.
  The BUILD files under `gen/go/**` are **hand-maintained**; the root `BUILD.bazel` sets
  `# gazelle:proto disable_global` so gazelle never generates conflicting proto rules. After adding a
  proto file, add its generated `.pb.go`/`.connect.go` to the existing `srcs` lists by hand, then run
  gazelle for the plain-Go packages.
- **Generated code is checked in** (`gen/go/**`, `web/src/gen/**`). Regenerate + commit; don't
  hand-edit generated files (the `.pb.go` descriptor is length-prefixed and will corrupt).
- **`cmd/` owns env parsing; internal packages take typed config.** Only `cmd/*` reads env vars
  (`caarlos0/env`); services receive a `Config` struct. `cmd/seed` is the exception (a driver binary).
- **Public vs protected is set in two places that must agree:** an RPC service is public iff it is
  registered *without* `tokenissuer.ConnectAuth(...)` in `internal/api/app.go` **and** marked
  `"public": true` for its path prefix in `services/gateway/gateway.json`. A single method cannot be
  public on an otherwise-protected service ŌĆö that is why `OnboardingService` is its own service.
- **Enum-like columns are stored as lowercase text** (`tier`/`status`/`billing_status`); the Go
  service layer maps text Ōåö proto enums (`internal/api/partner.go`). The DB layer
  (`internal/api/db/`) stays proto-free.
- **Graceful degradation:** API readiness must not depend on Postgres; handlers tolerate `db == nil`
  and return `CodeUnavailable`. Don't add readiness checks on transient DB availability.
- **Identity key:** an admin user is `app_users.id` (UUID), resolved from the OIDC
  `(external_issuer, external_subject)` pair via `authdb.UpsertUser`. That UUID becomes the token
  `subject` and threads into every authenticated call.
- **Auth signing keys must be stable** across replicas/restarts (`AUTH_PRIVATE_KEY_BASE64`,
  `AUTH_STATE_KEY_BASE64`); never derive them from process-random defaults in a shared deploy.
- **The gateway backend name/URL/issuer in `gateway.json` must match the k8s API Service**
  (`crew-demo-api`). A mismatch silently breaks routing and token validation.
